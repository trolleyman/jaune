# Annotated sample snapshots

These tests render real source files into a readable, reviewable form that makes jaune's
scope-labelling visible at a glance — and lets it be diffed directly against the canonical VS Code
tokenizer.

## Layout

| Path | What it is |
| --- | --- |
| `extra-input/` | Raw source samples **we** author, for cases the upstream corpus doesn't cover well (e.g. embedded HTML/Markdown). Input only — no annotations. |
| `jaune/` | Generated by **jaune**. One annotated file per input, tokenized by this crate. |
| `textmate-grammars-themes/` | Generated from the **reference** tokenizer (`vscode-textmate`, the engine VS Code and Shiki use). Same format, same formatter — so a diff against `jaune/` is a real tokenization difference. |
| `.reference-tokens/` | Intermediate per-sample token JSON emitted by the reference script. Git-ignored; regenerated on demand. |

The inputs are the upstream corpus in `../../textmate-grammars-themes/samples/*.sample` plus
everything in `extra-input/`. The grammar is taken from the first `.`-separated segment of the file
name: `bat.sample` → `bat`, `html.embedded.sample` → `html`.

## The format

Each file is the original source, still valid (if heavily annotated) code, with a header naming the
assumed root scope and — beneath every line — one comment per nested **scope region**:

```rust
// grammar: source.rust

use std::collections::HashMap;
//<0-------------------------- meta.use.rust
//<0 keyword.other.rust
//  ^-- entity.name.namespace.rust
//     ^- keyword.operator.namespace.rust
//       ^---------- entity.name.namespace.rust
```

Reading the markers:

- **Comments use the language's own syntax** (`//`, `#`, `--`, `;`, `rem`, `<!-- -->`, …), so the
  file still looks like real source. Embedded fixtures switch per block: CSS comments inside
  `<style>`, JS inside `<script>`, the fenced language inside a Markdown ```` ``` ```` block. The
  delimiter is best-effort for the long tail of languages and defaults to `#`; it is purely
  cosmetic and never affects the jaune-vs-reference diff.
- **The grammar root is assumed.** It is named once in the `grammar:` header and stripped from
  every line, instead of being repeated on every token.
- **`^` marks a span's start; `-` extends it.** `^----` is a five-column span. A bare `^` is one
  column.
- **Shared parent scopes are factored out.** Instead of repeating the full stack per token, each
  line shows only the scope added at that nesting depth, outermost span first. So a quoted string
  reads as one span for `string.quoted.double` with the punctuation/content deltas nested beneath:

  ```js
  "#go"
  // ^---- string.quoted.double.ts
  // ^ punctuation.definition.string.begin.ts
  //   ^^ variable.other.readwrite.ts
  //     ^ punctuation.definition.string.end.ts
  ```

- **`<n` instead of a caret** is used when a span starts *underneath* the comment opener, where a
  caret can't be drawn. The start column is written as `<n` and dashes run out to the span's end:
  `//<0------` is a span over columns 0–9. (`//^---` is preferred whenever the start clears the
  opener.)

## Regenerating

```sh
bun run scripts/reference-tokens.ts          # refresh the reference token JSON (needs `bun install`)
UPDATE_SNAPSHOTS=1 cargo test --test samples  # rewrite jaune/ and textmate-grammars-themes/
```

A plain `cargo test --test samples` re-renders and fails if anything diverges from the committed
snapshots, so unreviewed labelling changes can't slip through. The reference side is only checked
when `.reference-tokens/` is present, so CI without `bun` still runs the jaune side.
